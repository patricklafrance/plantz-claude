# ADR-0003: MSW + TanStack Query Data Layer

## Status

accepted

## Context

plantz-claude initially used TanStack DB with localStorage for plant data persistence. This approach had limitations: localStorage is not a realistic data layer for a production app, TanStack DB's `useLiveQuery` API does not exercise standard data fetching patterns (loading states, error handling, cache invalidation), and Storybook stories could not control data state without manipulating localStorage directly.

A migration was needed to adopt a data layer that mirrors real-world REST API patterns while still running entirely in the browser (no backend server).

## Options Considered

1. **MSW + TanStack Query** — Mock Service Worker intercepts `fetch()` calls to REST endpoints and serves from an in-memory database. TanStack Query provides `useQuery`/`useMutation` hooks with caching, optimistic updates, and loading/error states. Components use standard fetch-based patterns that would work with a real API.
2. **TanStack DB with localStorage** (status quo) — Client-side reactive database with localStorage persistence. Simple setup but unrealistic data flow, no loading/error states, and difficult to control in Storybook.
3. **json-server or similar local API server** — Run a real HTTP server during development. Adds process management complexity, requires starting a separate server alongside the dev server, and complicates Storybook and CI environments.

## Decision

Use MSW + TanStack Query (Option 1). MSW intercepts browser fetch requests at the service worker level, serving responses from an in-memory `Map<string, Plant>` database. TanStack Query manages server state with hooks that mirror production patterns. This combination provides realistic data fetching (loading states, errors, cache invalidation, optimistic updates) without requiring an external server process.

## Consequences

See the skill reference files (`tanstack-db.md` in `.claude/skills/plantz-sdlc-*/references/`) for the full data flow, hooks API, MSW setup, and Storybook integration patterns.

Additional implications:

- All domain modules depend on `@tanstack/react-query` as a peer dependency. The host app provides the `QueryClientProvider`.
- `@packages/plants-core` exposes MSW handlers and seed data via a `./msw` subpath export, keeping MSW-specific code separate from the main entry point.
- Every Storybook preview initializes MSW and wraps stories in `QueryClientProvider`. Per-story data variants use `parameters.msw.handlers` overrides.
- Data does not persist across page reloads — the in-memory DB resets each time. This is intentional for a proof-of-concept; a production app would point the same fetch calls at a real API.
- The `mockServiceWorker.js` service worker files are generated by MSW and must be present in each Storybook's and the host app's public directory.
