I want you to create a collection of agent skills to will map different profile of the Software Development Lifecycle. The goal is to build a collection of agent skills that will be used by an orchestor agent and different subagents to develop a feature.

All agent skills must be created using the `skill-creator` agent skill and 2 subagents to assist and challenge each others.

The agent skills to create are:

- `plantz-sdlc-orchestrator`: An agent skill for the orchestrator agent that will lead and be accountable for the development of the feature. The orchestrator agent will orchestrator the different steps of the process and coordinate subagents.
- `plantz-sdlc-plan`: An agent skill to plan the development of the code of the feature. The planning subagents will elaborate the plan the output it in a file.
- `plantz-sdlc-code`: An agent skill to execute the plan. The coding subagents will load the plan file and execute the plan.
- `plantz-sdlc-test`: An agent skill to test the code generated by the coding subagents. With the help of the orchestrator agent, the testing subagents will look for issues in the code and output their findings in a file.
- `plantz-sdlc-document`: An agent skill to update the repository agent documentation. Documentarian subagents will look at the plan and the changes and identify drifts in the existing documentation. If any drifts are found, they will fix them.
- `plantz-sdlc-merge`: An agent skill to commit the code and create and monitor a pull request.

## plantz-sdlc-orchestrator

The `plantz-sdlc-orchestrator` agent skill purpose is to lead the development of the feature, spawn subagents, and coordinate their work. This is the entry point for developing a new feature.

Through the whole process, this agent skill will do in the following order these steps (each step must be done before proceeding to the next one):

1. Generate an run uuid for this run, create the `./tmp/runs/[run-uuid]/` folder and pass the uuid to subagents.
2. Create a branch from.
3. Spawn two subagents that will use the `plantz-sdlc-plan` agent skill to plan the development of the feature. When the subagents are done, the orchestrator agent must look for `./tmp/runs/[run-uuid]/plan.md`.
4. Spawn two subagents that will use the `plantz-sdlc-code` agent skill to implement the plan. The plan must be provided by the orchestrator to the code subagents.
5. Spawn two subagents that will use the `plantz-sdlc-test` agent skill to test the generated code and iterate with the existing coding subagents to fix any issues found by the testing subagents. When the subagents are done, the orchestrator agent must look for `./tmp/runs/[hash]/test-issues-[iteration].md` files. If a file is found, the orchestrator agent must provide the issues to the existing code subagents to they fix them. Once fixed, the orchestrator must ask the existing test subagents to validate the changes again. This iteration should go on until there's no more issues to fix. Everytime an iteration is completed, the orchestrator should increment the iteration number.
6. Simplify the generated code using the `/simplify` skill.
7. Spawn two subagents that will use the `plantz-sdlc-document` agent skill to look for drifts between the code and agentic documentation and update if anything is found.
8. Spawn a subagent that will use the `plantz-sdlc-release` agent skill to commit the changes, open the pull request and monitor the pull request.
9. Clean up the `./tmp/runs/[run-uuid]/` folder.

This agent skill is the glue between every subagents that will take part in the SDLC.

When subagents rely on an `iteration` number, this number should be provided by this orchestrator.

### Run uuid

To create files without collisions, every run have it's own folder. This folder must be created at the beginning of the run by this agent. To avoid collision the name of the run foldr must be a uuid. The uuid will then be provided to different subagents that will need it to recover files in the run folder.

### Iteration number

In the same spririt as run uuid, some steps will iterate (for example the code and tests steps)... Those steps output files that must includes an iteration id to avoid name collision and content override. This agent must maintain the iteration number or provide it to subagents.

### Create a branch

Create a new branch from `main`. Use the conventional commit prefix matching the branch prefix (`feat:`, `fix:`, `chore:`, `docs:`, `refactor:`).

### /simplify

When the code is implemented and tested, it's important to optimize the code. To do this, use the native Claude Code `/simplify` skill.

## plantz-sdlc-plan

The `plantz-sdlc-plan` agent skill purpose is use plan mode to draft the technical approach to implement the feature. The goal is to output a `./tmp/runs/[run-uuid]/plan.md` file including the technical approach to implement the feature. The `run-uuid` will be provided by the orchestrator agent (or the consumer in this case).

The following instructions must be followed when drafting the plan:

- Before proceeding, look for relevant records in [ADR](./agent-docs/adr/index.md).
- Before proceeding, look for relevant records in [ODR](./agent-docs/odr/index.md).
- Before proceeding, read [ARCHITECTURE.md](./agent-docs/ARCHITECTURE.md).
- The code must be in React and TypeScript. To help with that, the agent must load the `workleap-react-best-practices` agent skill.
- The code must use Tailwind for styling.
- The code must be WCAG AA compliant. To help with that, the agent must load the `accessibility` agent skill.
- The code must support light and dark color scheme (when creating the skill, move the information from `./agent-docs/references/color-mode.md` to this agent skill and, if there's nothing left, delete the file).
- The code must support phone, tablet and desktop devices (when creating the skill, move the information from `./agent-docs/references/responsive-layout.md` to this agent skill and, if there's nothing left, delete the file).
- The code must use [Tanstack DB](https://tanstack.com/db/latest) for persistence. Currently, it should only support [local storage collections](https://tanstack.com/db/latest/docs/collections/local-storage-collection) (when creating the skill, move the information from `./agent-docs/references/tanstack-db.md` to this skill and, if there's nothing left, delete the file).
- The code must use [shadcn](https://ui.shadcn.com/) v4 with Base UI, Tailwind and PostCSS for components (when creating the skill, move the information from `./agent-docs/references/shadcn.md` to this skill and, if there's nothing left, delete the file).
- The agent must generate [Storybook](https://storybook.js.org/) stories with multiple variants for every components (when creating the skill, move the information from `./agent-docs/references/storybook.md` to theisskill and, if there's nothing left, delete the file).
- The agent must load the `frontend-design` agent skill to help design any feature.
- If the agent have to scaffold a new domain module, the agent must load and use the `plantz-scaffold-domain-module` agent skill to scaffold the domain.
- If the agent have to scaffold a new Storybook, the agent must load and use the `plant-zscaffold-domain-storybook` agent skill to scaffold the Storybook.

To execute this step of the SDLC, two subagents should be spawned to assist and challenge each others.

## plantz-sdlc-test

The `plantz-sdlc-test` agent skill purpose is to ensure that the generated code match the expected quality.

The following instructions must be followed when validating the generated code:

1. Read the generated `./tmp/runs/[run-uuid]/changes-[iteration].md` file. The `run-uuid` and `iteration` will be provided by the orchestrator agent (or the consumer in this case).
2. Execute `pnpm lint` from the root of the workspace
3. Load the `plantz-validate-modules` agent skill and validate that every module is conform.
4. Read the `./agent-docs/referenes/quality-gate.md` file for additional validation information (when creating the skill, move the relevant information from `./agent-docs/referenes/quality-gate.md` to this skill and, if there's nothing left, delete the file).
5. Load the `plantz-smoke-tests` agent skill to smoke tests every app.

When done, the validation subagents must write their findings in the `./tmp/runs/[hash]/test-issues-[iteration].md`.

To execute this step of the SDLC, two subagents should be spawned to assist and challenge each others.

## plantz-sdlc-document

The `plantz-sdlc-document` agent skill purpose is to ensure there's no drift between the agentic documentation (`./agent-docs`, `CLAUDE.md`) after a feature has been implemented.

The following instructions must be followed:

1. Read the generated `./tmp/runs/[run-uuid]/changes-[iteration].md` file. The `run-uuid` and `iteration` will be provided by the orchestrator agent (or the consumer in this case).
2. If there are changes affecting the [ADR](./agent-docs/adr.README.md), the orchestrator agent must update the ADR. When creating the skill, evaluate if the ADR template and README file could be move to the skill instead, and, if there's nothing left, delete the files.
3. If there are changes affecting the [ODR](./agent-docs/odr.README.md), the orchestrator agent must update the ODR. When creating the skill, evaluate if the ODR template and README file could be move to the skill instead, and, if there's nothing left, delete the files.
4. If there are changes affecting the [agent-docs reference files](./agent-docs/references), the orchestrator agent must update the references files. Don't forget to update the root CLAUDE.md index as well.

To execute this step of the SDLC, two subagents should be spawned to assist and challenge each others.

## plantz-sdlc-merge

The `plantz-sdlc-merge` agent skill purpose is to handle everything related to merging the code into main.

The steps are:

1. Commit the changes
2. Push and open a PR
3. Monitor the PR

### Commit

Commit the changes:

```
git add -A
git commit -m "<type>: <description>"
```

Use the conventional commit prefix matching the branch prefix (`feat:`, `fix:`, `chore:`, `docs:`, `refactor:`).

### Push and open PR

Push the commits and open a pull request:

```
git push -u origin <branch-name>
gh pr create \
  --title "<type>: <description>" \
  --body "$(cat <<'EOF'
## Changes
<description of the changes>

## Quality checks
- [ ] Lint
- [ ] Modules structure
- [ ] Quality gate
- [ ] Smoke tests

🤖 Generated with [Claude Code](https://claude.com/claude-code)
EOF
)"
```

Use the conventional commit prefix matching the branch prefix (`feat:`, `fix:`, `chore:`, `docs:`, `refactor:`).

The description of the changes must be obtained from an aggreation of all the `./tmp/runs/[hash]/changes-[iteration].md` files.

### Monitor PR

Monitor the pull request:

1. If any GitHub action workflows fails, spawn two subagents that will use the `plantz-sdlc-code` agent skill to fix the issues reported by the workflows and commit + push the changes.
2. If any comments is added to the pull request, spawn two subagent that will use `plantz-sdlc-code` and `plantz-sdlc-document` agent skills to evaluate if they are legitimate. If any are legitimate, ask to the 2 spawned subagents to fix the legitimate comments and commit + push the changes. Once resolved, resolve the comments in the pull request.
3. When all workflows except chromatic are green, and all pull request comments have been resolved, add the `run chromatic` label to the pull request. This will execute the Chromatic workflows.
4. If Chromatic workflows fails, tag the repository repository maintainers in the pull request and ask them to review Chromatic.
